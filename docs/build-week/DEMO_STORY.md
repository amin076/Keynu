# Keynu — OpenAI Build Week Demo Story

## Working Title

**Keynu — Helping AI Continue Real Software Engineering**

**Core Story:** Esbiko inspired Keynu. Today, Keynu helps improve Keynu itself.

## Scene 1 — Everything Started with Esbiko

**Visuals:** Show Esbiko and several interactive science simulations.

**Story:** I started by building Esbiko, an online science laboratory for teachers and students. At first, I used ChatGPT through the traditional generate, copy, paste, test, and repeat workflow.

**Main message:** Esbiko is the real project that revealed the need for Keynu.

## Scene 2 — The Project Kept Growing

**Visuals:** Show the Esbiko repository, simulation folders, assets and Git history.

**Story:** As Esbiko grew, it gained more simulations, components, assets and architectural decisions. Development continued across many sessions and became larger than a single AI conversation.

**Main message:** The challenge became maintaining engineering continuity across a long-running software project.

## Scene 3 — From Faster Coding to Persistent Engineering

**Visuals:** Show repository edits, terminal commands and multiple development sessions.

**Story:** Modern AI coding tools greatly accelerated software development by editing files and running commands directly. As the project continued to grow over many development sessions, it became clear that preserving mission context, architectural decisions and pending work across sessions was a separate engineering problem.

**Main message:** Faster code generation alone was not enough; persistent engineering continuity became the next challenge.

## Scene 4 — Keynu Was Created

**Visuals:** Show the BrowserAgent, KAP messages, repository memory and runtime drivers.

**Story:** Keynu began as a practical bridge between an AI conversation and a real development environment. It could receive structured jobs, operate on repository files, run tools, return verified reports and preserve mission state inside the project itself.

**Main message:** Keynu turns separate AI interactions into a continuous, repository-backed engineering workflow.

## Scene 5 — A Live KAP Job

**Visuals:** Show ChatGPT sending a small KAP job, Keynu executing it locally and returning a verified REPORT.

**Story:** In the live demo, the AI does not merely suggest code. It sends a structured job to Keynu. Keynu performs the requested repository operation, collects execution evidence and returns a verifiable result.

**Main message:** The conversation becomes connected to real, observable and recoverable software execution.

## Scene 6 — Keynu Became a Real Engineering Project

**Visuals:** Show the Keynu repository, BrowserAgent, mission system, runtime graph and dashboard.

**Story:** As Keynu gained more capabilities, it became a substantial software project itself. Browser reliability, mission recovery, verification, runtime intelligence and documentation all became real engineering challenges.

**Main message:** The tool created to support a complex project eventually became a complex project of its own.

## Scene 7 — Keynu Improves Keynu

**Visuals:** Show a Keynu mission reading the Keynu repository, applying a small change and returning a verified REPORT.

**Story:** The strongest proof of the system is dogfooding. Keynu now helps manage and improve its own codebase through small, recoverable jobs, persistent mission state and execution evidence.

**Main message:** Keynu is not only a prototype for other projects; it is already being used on the engineering problems that created it.

## Scene 8 — Tomorrow Morning

**Visuals:** Show the repository being reopened the next day and the mission resuming immediately.

**Story:** The next morning, the repository already contains the mission state, architectural decisions and remaining tasks. Instead of reconstructing the project from memory, development resumes from the last verified point.

**Main message:** Continuity is preserved inside the project, allowing engineering work to continue efficiently across separate sessions.

## Scene 9 — Beyond One Repository

**Visuals:** Show Keynu drivers connecting to files, browsers, development tools and external applications.

**Story:** The same mission-based runtime can support more than one codebase. Through explicit drivers and verified jobs, Keynu can connect AI reasoning to different applications and engineering environments without hiding execution behind an opaque agent loop.

**Main message:** Keynu is a runtime layer for persistent, tool-connected AI work across real projects.

## Scene 10 — The Build Week Proof

**Visuals:** Show the completed demo document beside the verified KAP reports that created it.

**Story:** This Build Week submission is itself produced through the Keynu workflow. The mission is divided into small recoverable steps, each executed locally and confirmed by evidence before the next step begins.

**Main message:** The demo does not only describe Keynu; the process of building the demo demonstrates how Keynu works.

## Final Message

Keynu turns disconnected AI conversations into persistent, verifiable engineering missions. It connects AI reasoning to real tools, preserves project continuity and makes every execution step inspectable.

For Build Week, the strongest evidence is the submission process itself: Keynu planned, executed, recovered and verified the work used to explain Keynu.
